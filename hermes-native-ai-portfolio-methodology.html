Hermes-Native AI Portfolio Methodology

Design outline for building a robust, multi-agent AI-powered portfolio system inside Hermes.
⚠ Status: DESIGN PHASE — methodology outline only; not yet implemented.

1. Design philosophy

Build a multi-agent portfolio system where each agent has a narrow, well-defined role, and orchestration enforces dependencies, quality gates, and auditability.

2. System architecture overview

A layered architecture: Human Operator → Orchestrator/Conductor → Specialist Analysts (macro, fundamental, technical/news) → Score Synthesizer → Portfolio Builder → Reviewer/Auditor. Each layer feeds the next via deterministic files in /opt/data/stock-research/data/processed/YYYY-MM-DD/.

3. Hermes profiles to create

Create purpose-built profiles instead of overloading default:

  • portfolio-orchestrator — plans rebalance and dispatches work.
  • macro-analyst — economic context and regime assessment.
  • fundamental-analyst — ratios, valuation, quality.
  • technical-analyst — momentum, sentiment, news.
  • scoring-synthesizer — combines signals into unified ranks.
  • portfolio-builder — applies constraints and risk overlays.
  • portfolio-reviewer — final quality gate and sanity checks.
  • execution-clerk — trade sheets and logs (only after paper-trading approval).

4. Robust data pipeline

Cache all raw data under dated folders. A data-validator subagent checks non-empty files, valid tickers, and cross-source consistency before scoring runs. If validation fails, the orchestrator blocks the task rather than proceeding with garbage data.

5. Scoring workflow

Use structured prompt templates saved as Hermes skills. Every analyst outputs JSON with score, confidence, thesis, bull_case, bear_case, key_risks, and data_cutoff. Run macro, fundamental, and technical scores in parallel via delegate_task or Kanban workers; the synthesizer reconciles divergences and produces a final rank.

6. Portfolio construction

The portfolio-builder selects the top N assets and applies constraints (max position, sector caps, min/max holdings) plus risk overlays (volatility targeting, correlation limits, drawdown guards). Weight math is done in Python via execute_code, not by the LLM, to guarantee accuracy. Output is recommendation.json plus a trade-sheet CSV.

7. Quality gates

The reviewer runs a checklist: explicit data cutoff, cited sources or 'unknown' markers, constraint satisfaction, documented bear case, weights sum to 100%, no duplicates/delisted symbols, month-over-month change explanation, and risk disclaimer. Any failure triggers a kanban_block with specific reason.

8. Durable execution via Kanban

Convert monthly rebalance into a Kanban board with parent-linked tasks so it survives crashes. Use parents=[...] to enforce dependency order and goal_mode=True for long-running data-fetch tasks. The dispatcher auto-spawns the next stage only after predecessors complete.

9. Scheduling

Use hermes cron (e.g., 0 9 1 * *) to trigger rebalances. Deliver completion notifications to the gateway home channel. Provide a manual override Kanban card for emergency rebalances.

10. Memory and auditability

Store user preferences, lessons learned, and prompt versions in Hermes memory. Maintain a file-system audit trail per rebalance: inputs.json, raw data, scores.json, recommendation.json, rationale.md, trade-sheet.csv, and reviewer-signoff.md.

11. Robustness patterns

Agent-level: always kanban_show first, never invent data, use goal_mode, set timeouts, limit delegation depth. Pipeline-level: redundant data sources, deterministic paths, JSON schema validation, idempotent stages. Human-level: approval gates, gateway notifications, reviewer blocks on terminal actions.

12. Backtesting and paper-trading

Before live capital, run a backtest track replaying historical dates and a paper-trade track producing trade sheets without execution. Only connect the execution-clerk to a broker API after acceptable Sharpe, drawdown, and turnover metrics.

13. Implementation phases

Phase 1: single asset/single signal. Phase 2: multi-asset parallel scoring. Phase 3: durable Kanban rebalance with reviewer. Phase 4: cron + notifications + HTML reports. Phase 5: paper trading. Phase 6: optional live execution.

16. Disclaimer

This is a research and educational design outline. It is not investment advice. Any live implementation requires legal, compliance, and risk review, plus thorough backtesting and paper trading.
